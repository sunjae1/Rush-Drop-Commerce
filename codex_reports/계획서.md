# 기존 수동 배포 -> GitHub Actions 자동화 보고서

## 1. 전제

이 보고서는 아래 상태를 전제로 한다.

- S3 버킷은 이미 있다.
- CloudFront 배포도 이미 있다.
- 지금도 수동 배포는 된다.
- 지금 부족한 것은 자동화만이다.

즉, 이번 작업은 AWS 인프라를 새로 만드는 작업이 아니라,  
지금 손으로 하던 배포를 GitHub Actions가 대신하게 만드는 작업이다.

---

## 2. 현재 프로젝트 기준으로 확인된 점

- 저장소 기준 경로는 `shopping_fe` 이다.
- 빌드 명령어는 `npm run build` 이다.
- 배포 대상 폴더는 `dist/` 이다.
- 현재 `.env.production` 은 아래 의미로 쓰고 있다.

```dotenv
VITE_API_BASE_URL=
VITE_USE_DEMO_DATA=false
```

중요한 점:

- 지금 운영 방식은 `VITE_API_BASE_URL` 을 별도 도메인으로 넣는 방식이 아니다.
- 주석상 CloudFront behavior에서 `/api` 를 백엔드로 보내는 구조다.
- 그래서 GitHub Actions에서 프론트 운영 환경변수를 많이 넣을 필요가 없다.
- 현재 코드 기준으로는 `VITE_API_BASE_URL` 이 없어도 기본값이 `""` 이고,
  `VITE_USE_DEMO_DATA` 가 없어도 기본적으로 `false` 처럼 동작한다.

즉, 이번 자동화의 핵심은 프론트 env 설계가 아니라:

1. GitHub Actions가 빌드하고
2. S3에 업로드하고
3. CloudFront invalidation 하는 흐름을 붙이는 것이다.

---

## 3. 내가 해야 할 일 순서

### 1. 지금 수동 배포에서 쓰는 정확한 값 3개를 먼저 확정한다

이 3개는 무조건 먼저 필요하다.

- AWS 리전
- S3 버킷 이름
- CloudFront Distribution ID

예시:

```text
AWS_REGION=ap-northeast-2
S3_BUCKET=내-프론트-버킷명
CLOUDFRONT_DISTRIBUTION_ID=E123456789ABCDE
```

주의:

- 지금 수동 배포에서 버킷 루트에 올리는지, 특정 prefix에 올리는지도 같이 확인한다.
- 예를 들어 `s3://bucket/` 에 올리는지, `s3://bucket/prod/` 에 올리는지 확인해야 한다.
- 자동화는 지금 수동 방식과 똑같이 맞추는 게 가장 안전하다.

---

### 2. GitHub Actions가 AWS에 접근할 자격 증명을 준비한다

이번 1차 자동화는 가장 단순한 방식으로 가는 것이 낫다.

추천:

- Access Key 방식으로 먼저 붙인다.

필요한 것:

- 배포용 IAM User 또는 이미 수동 배포에 쓰는 전용 IAM 자격
- 그 자격에 아래 권한 부여

필수 권한:

- `s3:ListBucket`
- `s3:PutObject`
- `s3:DeleteObject`
- `s3:GetObject`
- `cloudfront:CreateInvalidation`

가능하면 권한 범위는:

- 해당 S3 버킷
- 해당 CloudFront 배포

로만 제한한다.

참고:

- 장기적으로는 OIDC가 더 좋지만,
- 지금 목표가 "수동 배포를 빨리 자동화" 라면 Access Key가 가장 빠르다.

---

### 3. GitHub Secrets와 Variables를 등록한다

#### GitHub Secrets에 넣을 것

- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`

#### GitHub Variables에 넣을 것

- `AWS_REGION`
- `S3_BUCKET`
- `CLOUDFRONT_DISTRIBUTION_ID`

#### 현재 프로젝트 기준으로 굳이 안 넣어도 되는 프론트 변수

- `VITE_API_BASE_URL`
- `VITE_USE_DEMO_DATA`

이유:

- 현재 코드 기본값이 이미 운영과 맞기 때문이다.
- 지금 `.env.production` 내용도 사실상 기본값과 동일하다.

정리하면 이번 자동배포 1차 구성에서  
GitHub에 꼭 넣어야 하는 값은 사실상 AWS 관련 값들이다.

---

### 4. 배포 브랜치를 정한다

예를 들어:

- `main` 에 push되면 자동배포

가장 단순한 시작 방식:

1. 개발은 다른 브랜치에서 작업
2. 배포할 때만 `main` 으로 merge
3. `main` push 시 자동 배포

---

### 5. GitHub Actions workflow 파일을 만든다

만들 파일:

- `.github/workflows/deploy-fe.yml`

이 workflow가 할 일은 아래 6단계면 충분하다.

1. 저장소 checkout
2. Node 설치
3. `npm ci`
4. `npm run build`
5. `dist/` 를 S3에 업로드
6. CloudFront invalidation 실행

핵심 명령은 보통 아래 형태다.

```bash
aws s3 sync dist s3://$S3_BUCKET --delete
aws cloudfront create-invalidation --distribution-id $CLOUDFRONT_DISTRIBUTION_ID --paths "/*"
```

만약 지금 수동 배포가 버킷 root가 아니라 특정 경로라면:

```bash
aws s3 sync dist s3://$S3_BUCKET/특정경로 --delete
```

처럼 지금 방식 그대로 맞춰야 한다.

---

### 6. 첫 테스트 push를 한다

첫 테스트 때 확인할 것:

1. GitHub Actions가 실행되는지
2. `npm ci` 가 성공하는지
3. `npm run build` 가 성공하는지
4. S3 업로드가 성공하는지
5. CloudFront invalidation이 성공하는지

이 단계에서 실패하면 대개 원인은 아래 중 하나다.

- GitHub Secret 이름 오타
- S3 버킷명 오타
- Distribution ID 오타
- IAM 권한 부족
- 실제 수동 배포 경로와 workflow 경로가 다름

---

### 7. 배포 결과를 직접 확인한다

확인 순서:

1. CloudFront 주소 접속
2. 메인 페이지가 최신 버전인지 확인
3. 새로고침 확인
4. 라우팅 페이지 직접 진입 확인
5. `/api` 요청이 기존처럼 정상 동작하는지 확인

중요:

- 이번 작업은 인프라 변경이 아니라 자동화 추가이므로,
- 배포 결과는 "수동 배포 때와 완전히 같아야" 정상이다.

---

## 5. 이번 단계에서 하지 않아도 되는 것

이번 자동화 작업에서 굳이 다시 손대지 않아도 되는 것:

- S3 버킷 새로 만들기
- CloudFront 새로 만들기
- API 도메인 구조 다시 설계하기
- CORS를 처음부터 다시 잡기
- `.env.production` 을 GitHub에 업로드하기

지금 목표는 새 구조 설계가 아니라  
기존 수동 배포 절차를 GitHub Actions로 옮기는 것이다.

---

## 6. 최종 작업 순서 요약

1. 현재 수동 배포에 쓰는 `AWS_REGION`, `S3_BUCKET`, `CLOUDFRONT_DISTRIBUTION_ID` 확인
2. 배포용 AWS Access Key 준비
3. GitHub Secrets 등록
4. GitHub Variables 등록
5. `.github/workflows/deploy-fe.yml` 작성
6. 배포 브랜치에 push
7. Actions 로그 확인
8. CloudFront에서 실제 결과 확인
