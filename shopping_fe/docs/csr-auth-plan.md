# CSR 인증/로그인 개선 계획 및 Q&A

## 구현 상태

2026-03-11 기준 아래 항목은 반영했습니다.

- 보호 라우트 도입
- `returnTo` 기반 로그인 복귀
- 보호 API 401 시 로그인 페이지 이동
- 로그인 실패 배너/입력폼 문구 추가
- `POST /api/logout` 추가
- 로그아웃 후 홈(`/`) 이동

## 1. 현재 판단

현재 React 로그인 문제가 **세션 기반이라서 원천적으로 불가능한 문제**는 아닙니다.

CSR에서도 세션 로그인은 충분히 가능합니다.

지금 더 큰 문제는 아래입니다.

- 프런트에 `401 -> 어떤 UX를 줄지`에 대한 공통 규칙이 약함
- 보호 페이지와 공개 페이지의 동작 차이가 명확히 정리되어 있지 않음
- 로그아웃이 CSR 친화적인 API 없이 SSR 구조에 일부 기대고 있음
- 백엔드 401 응답은 존재하지만, 프런트가 그 응답을 보고 어떻게 움직일지는 아직 통일되지 않음

즉, 지금 우선순위는 `JWT 전환`보다 `세션 기반 CSR 인증 흐름 정리`입니다.

## 2. 정정: "전역 정책이 왜 없어? ApiExceptionHandler 있잖아"

이 지적이 맞습니다.  
정확히 나누면 아래처럼 봐야 합니다.

### 백엔드 전역 정책

부분적으로 있습니다.

- `ApiExceptionHandler`는 API 예외를 HTTP 응답으로 정리합니다.
- `ApiLoginFailureHandler`는 `/api/login` 실패 시 401과 메시지를 내려줍니다.

하지만 이건 **서버가 어떤 HTTP 응답을 줄지**를 정하는 층입니다.

### 프런트 전역 정책

이건 별개입니다.

프런트 전역 정책은 이런 것입니다.

- 401을 받으면 무조건 로그인 페이지로 갈지
- 어떤 페이지에서는 문구만 보여줄지
- `returnTo`를 붙일지
- 보호 페이지 접근 시 라우트 단계에서 막을지
- 로그인 후 원래 보려던 페이지로 복귀시킬지

즉, **백엔드 전역 예외 처리**와 **프런트 전역 인증 UX 정책**은 다른 문제입니다.

## 3. "API에서 401 던지면 프런트에서 redirect 하는 거 아니야?"

자동으로는 아닙니다.

왜냐하면 React CSR에서 API 호출은 보통 `fetch`로 이루어지기 때문입니다.

### SSR일 때

브라우저가 페이지를 직접 요청합니다.

예:

- 사용자가 `/mypage` 접속
- 서버가 세션 확인
- 로그인 안 되어 있으면 서버가 `/login`으로 redirect 응답
- 브라우저가 실제 페이지 이동

즉, 서버 응답이 곧 화면 이동으로 이어집니다.

### CSR일 때

브라우저는 이미 React 페이지 안에 들어와 있습니다.

예:

- 사용자가 React에서 장바구니 버튼 클릭
- JS가 `fetch('/api/cart')` 호출
- 서버가 401 반환

이 경우 브라우저는 **그냥 fetch 요청이 실패했다고만 알지**,  
현재 보고 있는 화면을 자동으로 `/login`으로 바꾸지 않습니다.

즉, **401 응답을 받은 뒤 `navigate('/login')` 같은 처리는 프런트 코드가 직접 해야 합니다.**

## 4. 이번 요청 기준 확정 정책

이 정책으로 진행하면 됩니다.

### 로그인 성공

- 기본은 이전 경로 `returnTo`가 있으면 그곳으로 이동
- 없으면 `/account`

### 로그인 실패

- 페이지 상단 배너 표시
- 입력폼 근처에도 실패 문구 표시

### 세션 만료(401)

- 보호 API 호출이면 로그인 페이지로 이동
- `로그인이 필요합니다.` 문구와 함께 `returnTo` 유지

### 공개 API 401

- 전역 redirect 하지 않음
- 페이지 컨텍스트에 맞게 처리

### 로그아웃

- 백엔드에 `/api/logout` 추가
- 로그아웃 성공 후 홈(`/`)으로 이동

### 현재 사용자 확인용 API

- `/api/me` 또는 `/api/session`은 **이번 요청에서 구현하지 않음**
- 다음 요청에서 고려할 제안 사항으로만 유지

## 5. "페이지 컨텍스트에 맞게 처리"가 무슨 뜻인가

이건 **같은 401이어도 모든 화면에서 똑같이 로그인 페이지로 보내지 말자**는 뜻입니다.

### 예시 1. 보호 기능

- 장바구니 조회
- 주문하기
- 마이페이지 진입
- 게시글 작성
- 댓글 작성/수정/삭제

이건 로그인 없이는 의미가 없습니다.  
그래서 401이면:

- 로그인 페이지로 이동
- `returnTo` 저장
- `로그인이 필요합니다.` 안내

### 예시 2. 공개 화면 안의 선택적 개인화 정보

- 홈 화면에서 "현재 로그인 사용자 이름"만 가져오는 요청
- 공개 커뮤니티 목록 화면에서 로그인 사용자 뱃지 표시

이건 실패해도 페이지 전체를 막을 필요는 없습니다.

그래서 401이면:

- 로그인 사용자 정보만 비워서 렌더
- "로그인 후 더 많은 기능 이용 가능" 정도의 안내만 표시
- 전체 페이지 redirect는 하지 않음

즉,

- **기능 자체가 보호 대상이면 redirect**
- **페이지 자체는 공개인데 일부 개인화 데이터만 실패한 거면 문구만 처리**

이 의미입니다.

## 6. "API에서 401 응답은 다 있지 않아? 뭐가 부족한 거야?"

401이 **존재하는 것**과 **프런트가 쓰기 좋게 충분한 것**은 다릅니다.

현재 부족한 점은 아래입니다.

### 1. 응답 바디가 일관되지 않음

예:

- 어떤 곳은 401 + 빈 body
- 어떤 곳은 401 + `"로그인이 필요합니다."`
- 로그인 실패는 `ApiLoginFailureHandler`에서 별도 문자열 반환

즉, 프런트 입장에서는 항상 같은 구조로 읽기 어렵습니다.

### 2. 401 발생 지점의 의미가 섞여 있음

예:

- `/api`는 security 설정상 `permitAll`인데 컨트롤러 내부에서 직접 401 반환
- `/api/cart` GET은 비로그인이어도 200 + 빈 장바구니처럼 동작
- 다른 보호 API는 401 반환

즉, "비로그인이면 항상 401" 같은 단순 규칙이 안 됩니다.

### 3. machine-readable code가 없음

지금은 대부분 사람이 읽는 문자열 위주입니다.

예를 들면 이런 값이 없습니다.

- `AUTH_REQUIRED`
- `LOGIN_FAILED`
- `SESSION_EXPIRED`

이런 코드가 있으면 프런트가 분기하기 더 쉽습니다.

### 4. redirect 정책은 원래 응답만으로 완성되지 않음

서버가 401을 잘 보내도,

- 어디로 이동할지
- 언제는 이동하지 않을지
- 원래 페이지를 저장할지

이건 결국 프런트 정책이 필요합니다.

## 7. `/api/logout`은 왜 추가하는가

이 항목은 현재 구현 완료 상태입니다.  
이제 프런트는 `POST /api/logout`을 직접 호출하고, 백엔드는 세션을 종료한 뒤 프런트는 홈(`/`)으로 이동합니다.

이건 동작은 할 수 있어도 CSR 친화적이지 않습니다.

원하는 방향은 아래입니다.

- 프런트가 `/api/logout` 호출
- 백엔드가 세션 invalidate
- 프런트는 홈(`/`)으로 이동

이렇게 되면:

- SSR 템플릿 구조 의존 제거
- 로그아웃 플로우 단순화
- 프런트 코드 안정성 상승

## 8. 이번 구현 범위 계획

승인 후 구현할 범위는 아래로 잡는 것이 좋습니다.

### Phase 1. 프런트 인증 UX 정책 고정

- `returnTo` 규칙 추가
- 로그인 성공/실패 메시지 규칙 추가
- 세션 만료 문구 규칙 추가

### Phase 2. 보호 라우트 도입

- `/cart`
- `/account`
- 주문 실행
- 게시글 작성
- 댓글 작성/수정/삭제

이 경로/행동에 대해 로그인 가드 추가

### Phase 3. 공통 401 처리 정리

- API client에서 401 식별
- 보호 API에서 401이면 로그인 페이지로 이동
- 공개 화면의 선택적 데이터는 redirect 없이 로컬 처리

### Phase 4. 로그인 페이지 보강

- 상단 배너
- 필드 근처 실패 문구
- `returnTo` 복귀
- 이미 로그인 상태면 적절한 위치로 이동

### Phase 5. 백엔드 `/api/logout` 추가

- 세션 종료용 API 추가
- 프런트는 이 엔드포인트 호출 후 홈(`/`)으로 이동

### Phase 6. 테스트

- 로그인 성공 후 `returnTo` 이동
- 로그인 실패 문구 표시
- 보호 라우트 접근 시 로그인 페이지 이동
- 401 발생 시 보호/공개 화면 동작 차이 검증
- 로그아웃 후 홈 이동 검증

## 9. `/api/me` 또는 `/api/session`에 대한 판단

이건 `mypage` 얘기와는 다릅니다.

`/api/myPage`는 화면 데이터가 큽니다.

- user
- orders
- posts
- cart

반면 `/api/me` 또는 `/api/session`은 보통 아주 가볍게:

- 지금 로그인했는지
- 로그인 사용자 이름/이메일이 뭔지

이 정도만 확인하는 용도입니다.

하지만 요청하신 대로 **이번 요청에서는 구현 범위에 넣지 않고 제안 사항으로만 유지**합니다.

## 10. SSR과 CSR에서 인증 흐름이 다른 이유

이 부분을 가장 쉽게 설명하면 아래입니다.

### SSR

서버가 HTML까지 만들어서 보냅니다.

흐름:

1. 브라우저가 페이지 URL 요청
2. 서버가 세션 확인
3. 로그인 안 되어 있으면 서버가 바로 `/login`으로 redirect
4. 브라우저가 그 페이지로 이동

즉, **페이지 이동의 주도권이 서버에 큽니다.**

### CSR

처음 HTML은 거의 껍데기고, 실제 화면 전환과 데이터 요청은 JS가 담당합니다.

흐름:

1. 브라우저가 React 앱 로드
2. 이후 화면 데이터는 `fetch('/api/...')`
3. 서버가 401 반환
4. JS가 그걸 보고 직접 redirect 또는 문구 표시를 결정

즉, **화면 이동의 주도권이 프런트에 큽니다.**

## 11. "CSR은 브라우저가 아닐 수도 있어서 JWT를 쓰는 거야?"

반은 맞고, 반은 다릅니다.

정확히는:

- CSR도 보통 브라우저에서 실행됩니다.
- 그래서 `JSESSIONID` 쿠키도 보낼 수 있습니다.
- CSRF 토큰도 처리할 수 있습니다.

즉, **CSR이라서 세션/쿠키를 못 쓰는 건 아닙니다.**

### 세션 + CSRF 방식

특징:

- 브라우저가 쿠키를 자동 전송
- 서버 세션에 로그인 상태 저장
- 상태 변경 요청에서는 CSRF 방어 필요

장점:

- 서버 주도 보안 모델이 단순
- Spring Security 기본 구조와 잘 맞음

불편한 점:

- CSR에서는 CSRF 토큰을 프런트가 별도로 받아야 할 수 있음
- cross-origin 배포 시 쿠키 정책이 까다로워짐

### JWT 방식

특징:

- 보통 `Authorization: Bearer ...` 헤더로 토큰 전달
- 서버가 세션을 안 들고 가는 무상태 구조에 잘 맞음

장점:

- 모바일 앱, 외부 클라이언트, 완전 분리 배포에 유리
- same-origin 세션보다 배포 구조가 자유로울 수 있음

불편한 점:

- access token / refresh token 설계 필요
- 만료 처리 복잡
- 저장 위치에 따라 XSS/보안 고민 필요

즉, JWT는 **CSR이라서 무조건 써야 하는 것**이 아니라  
**클라이언트 종류와 배포 구조가 다양해질 때 더 매력적인 선택지**입니다.

## 12. 최종 권장안

현재는 아래 순서가 맞습니다.

1. 세션 기반 유지
2. 프런트 401 처리 정책 정리
3. 보호 라우트 추가
4. 로그인 실패/세션 만료 UX 추가
5. `/api/logout` 추가
6. 로그아웃 후 홈 이동

JWT 전환은 그 다음 아키텍처 단계에서 별도로 검토하는 것이 맞습니다.
