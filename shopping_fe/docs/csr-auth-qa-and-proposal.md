# CSR 인증 Q&A 및 제안 정리

## 1. "보호 기능들은 결국 다 같은 로그인 페이지로 보내는 거 아니야?"

맞습니다.

보호 기능끼리는 결국 같은 로그인 페이지로 보내는 것이 맞습니다.

즉, 아래 기능들은 모두 같은 정책입니다.

- 장바구니 조회
- 주문하기
- 마이페이지 진입
- 게시글 작성
- 댓글 작성/수정/삭제

이 기능들은 로그인 없이는 의미가 없으므로:

- 로그인 페이지로 이동
- `returnTo` 저장
- `로그인이 필요합니다.` 안내

여기서 제가 말한 "같은 401이어도 모든 화면에서 똑같이 보내지 말자"의 핵심은
**보호 기능들끼리 구분하자는 뜻이 아니라, 보호 화면과 공개 화면을 구분하자는 뜻**입니다.

정리:

- 보호 기능 401: 로그인 페이지로 보냄
- 공개 화면 안의 선택적 데이터 401: 문구만 처리하고 페이지 자체는 유지 가능

## 2. "`GET /api/cart`도 로그인 필요로 바꾸자"

이 방향이 더 일관적입니다.

이유:

- 장바구니는 보호 기능입니다.
- 비로그인 상태에서 `200 + 빈 장바구니`를 주면
  - 실제로 비어 있는 장바구니인지
  - 로그인 안 해서 못 보는 건지
  - 세션이 만료된 건지
  구분이 흐려집니다.

따라서 권장 정책은:

- `GET /api/cart`
- `POST /api/cart/items/{itemId}`
- `DELETE /api/cart/items/{itemId}`

모두 비로그인 시 `401`로 통일

## 3. `AUTH_REQUIRED`, `LOGIN_FAILED`, `SESSION_EXPIRED`는 어디에 넣는가?

보통 **response body의 JSON field**에 넣습니다.

예:

```json
{
  "code": "AUTH_REQUIRED",
  "message": "로그인이 필요합니다."
}
```

또는:

```json
{
  "code": "LOGIN_FAILED",
  "message": "이메일 또는 비밀번호를 확인해주세요."
}
```

즉, 사람이 읽는 `message`와 프런트가 분기하는 `code`를 같이 두는 방식입니다.

### 왜 필요한가

문자열 메시지만 있으면 프런트가:

- 언어가 바뀌었을 때
- 문구가 약간 수정됐을 때
- 서로 다른 401 사유를 나눠야 할 때

분기하기 어려워집니다.

`code`가 있으면 프런트는 안정적으로 분기할 수 있습니다.

### `SESSION_EXPIRED`는 주의점

이 코드는 서버가 진짜로 "만료"를 구분할 수 있을 때만 정확합니다.

세션 기반에서는 서버 입장에서 아래 둘이 비슷하게 보일 수 있습니다.

- 애초에 로그인 안 함
- 예전에 로그인했지만 세션 만료

그래서 현실적으로는:

- `AUTH_REQUIRED`만 먼저 도입
- 나중에 필요하면 `SESSION_EXPIRED` 구분 추가

이 순서가 보통 안전합니다.

## 4. "CSR에서도 CSRF 토큰을 백엔드가 프런트에 주면 되는 거 아니야?"

네, 맞습니다.

핵심은 **줄 수 있느냐가 아니라, 어떤 방식으로 줄 것이냐**입니다.

가능한 방식은 여러 가지입니다.

### 방식 1. SSR HTML에 hidden input으로 넣기

이건 기존 서버 렌더링 방식에서 흔합니다.

예:

```html
<input type="hidden" name="_csrf" value="..." />
```

장점:

- 서버 템플릿과 잘 맞음

단점:

- React CSR에서는 매 요청마다 이 HTML 구조에 기대면 번거롭고 결합도가 높아짐

### 방식 2. meta tag로 내려주기

예:

```html
<meta name="_csrf" content="...">
```

프런트 JS가 읽어서 요청 헤더에 넣을 수 있습니다.

### 방식 3. `/api/csrf` 같은 전용 엔드포인트

예:

```json
{
  "headerName": "X-CSRF-TOKEN",
  "token": "..."
}
```

CSR에서는 이 방식이 제일 명확한 편입니다.

### 방식 4. 쿠키로 전달

백엔드가 CSRF 토큰을 쿠키에 넣고,
프런트가 그 값을 읽어 헤더에 다시 넣는 방식도 있습니다.

## 5. 그럼 왜 "CSR에서는 CSRF가 까다롭다"고 하나?

SSR에서는 서버가 페이지를 만들 때 이미 토큰을 HTML 안에 자연스럽게 넣을 수 있습니다.

CSR에서는 보통:

1. 프런트 앱은 정적 HTML/JS로 먼저 뜸
2. 이후 API 호출은 JS가 담당
3. 상태 변경 요청마다 CSRF 토큰을 별도로 챙겨야 할 수 있음

즉, **토큰을 전달하는 추가 설계가 필요해서 까다롭다**는 뜻입니다.

불가능하다는 뜻은 아닙니다.

## 6. "cross-origin 배포 시 쿠키 정책이 까다롭다"는 뜻

예를 들어:

- 프런트: `http://localhost:5173`
- 백엔드: `http://localhost:8080`

이렇게 출처(origin)가 다르면 쿠키를 다루는 조건이 까다로워집니다.

세션 기반에서는 브라우저가 `JSESSIONID` 쿠키를 보내줘야 합니다.

그런데 cross-origin에서는 보통 아래 조건이 맞아야 합니다.

- 프런트 fetch에 `credentials: "include"`
- 백엔드 CORS에서 credentials 허용
- 쿠키의 `SameSite` 정책 설정
- 실제 HTTPS 환경이면 `Secure` 설정까지 고려

즉, "세션이 안 된다"가 아니라
**쿠키를 안전하게 오가게 만드는 설정이 더 많다**는 뜻입니다.

## 7. `SameSite` 때문에 왜 복잡해지나

브라우저는 외부 사이트에서 자동으로 쿠키를 보내는 걸 제한하려고 합니다.

이건 CSRF 공격을 줄이기 위한 정책입니다.

예를 들어 다른 사이트가 내 서버로 악성 POST 요청을 보내도,
세션 쿠키가 자동으로 붙으면 위험해질 수 있습니다.

그래서 브라우저는 쿠키를 아무 상황에서나 보내지 않도록 제한합니다.

세션 기반 + cross-origin 환경에서는 이 정책을 정확히 맞춰야 합니다.

## 8. "저장 위치에 따라 XSS/보안 고민 필요"는 무슨 뜻인가

이 말은 주로 JWT 이야기에서 나옵니다.

### JWT를 `localStorage`에 저장하는 경우

장점:

- 프런트에서 꺼내 쓰기 쉬움

단점:

- 만약 XSS가 나면 악성 스크립트도 `localStorage`를 읽을 수 있음
- 그러면 토큰이 탈취될 수 있음

### JWT를 `sessionStorage`에 저장하는 경우

비슷한 장단점이 있습니다.

- 브라우저 탭 단위로는 덜 오래가지만
- 여전히 JS에서 읽을 수 있으므로 XSS에 취약

### JWT를 HttpOnly 쿠키에 저장하는 경우

장점:

- JS가 직접 읽지 못함
- 그래서 토큰 탈취 측면에서는 더 안전할 수 있음

단점:

- 쿠키 기반이 되므로 다시 CSRF 고려가 필요

즉,

- `localStorage` 계열: XSS에 더 약함
- HttpOnly 쿠키: XSS 탈취에는 강하지만 CSRF 고려 필요

그래서 "JWT 쓰면 끝"이 아니라 저장 위치까지 같이 설계해야 합니다.

## 9. 지금 프로젝트 기준 권장안

지금은 JWT 전환보다 아래 순서가 더 적절합니다.

### 우선순위

1. 세션 기반 유지
2. 보호 기능 401 정책 통일
3. `GET /api/cart`도 보호 API로 통일
4. 응답 body에 `code`, `message` 구조 도입
5. 로그인 페이지 `returnTo` 복귀 정책 유지
6. `/api/logout` 사용

### 응답 code 제안

- `AUTH_REQUIRED`
- `LOGIN_FAILED`
- `ACCESS_DENIED`

`SESSION_EXPIRED`는 나중에 필요하면 추가 검토

## 10. 승인 후 구현 제안 범위

승인하시면 아래 범위로 가는 것이 좋습니다.

### 백엔드

- `GET /api/cart` 비로그인 시 `401`
- 401/403 응답의 `code`, `message` 구조 통일
- 로그인 실패 시 `LOGIN_FAILED`
- 인증 필요 시 `AUTH_REQUIRED`

### 프런트

- 보호 기능 401이면 로그인 페이지로 이동
- `returnTo` 유지
- 로그인 성공 시 원래 페이지 복귀
- 공개 화면은 redirect 없이 문구/상태만 처리

## 11. 최종 한 줄 요약

지금은:

- **보호 기능은 모두 로그인 페이지로 보내는 게 맞고**
- **공개 화면 안의 선택적 데이터는 굳이 다 보내지 않는 게 핵심**

그리고 JWT 전환보다 먼저,

- `GET /api/cart` 보호화
- 응답 `code/message` 정리
- 세션 기반 CSR 흐름 정리

이 순서가 맞습니다.
