# 백엔드 미연결 시 화면 소실 대응 보고서

## 1. 문제 요약

- 증상: 프런트를 열면 화면이 잠깐 보였다가 곧 사라진다.
- 재현 조건: 백엔드 서버가 꺼져 있거나 `/api` 요청이 네트워크 단계에서 실패한다.

## 2. 원인 분석

- `SessionProvider`는 첫 마운트에서 `fetchSession()`으로 세션을 확인한다.
- 백엔드가 죽어 있으면 `fetch('/api')`가 reject되고, 이 실패가 초기화 경로에서 잡히지 않았다.
- 같은 패턴이 `RequireAuth`, `CartProvider`의 `useEffect` 비동기 호출에도 있었다.
- 공개 화면은 자체적으로 오류를 배너로 처리하고 있었지만, 세션 초기화 경로는 전역 미처리 rejection으로 번질 수 있었다.

## 3. 구현 계획

1. 초기 세션 조회 실패를 `SessionProvider` 안에서 상태로 흡수한다.
2. 앱 쉘은 유지하고, 사용자에게 백엔드 연결 실패 메시지를 보여준다.
3. `useEffect`에서 fire-and-forget으로 호출하던 비동기 함수들의 미처리 rejection을 막는다.
4. 백엔드 미연결 재현 테스트를 추가해 회귀를 방지한다.

## 4. 적용 내용

- `SessionContext`
  - `sessionError` 상태 추가
  - 네트워크 실패 시 세션을 guest 상태로 정리하고 오류 메시지 저장
  - 초기 `refreshSession()` 호출에 `.catch(() => undefined)` 추가
- `AppShell`
  - 세션 오류 배너 렌더 추가
- `RequireAuth`
  - 조용한 세션 재검증 실패가 미처리 rejection으로 남지 않도록 보강
- `CartContext`
  - 사용자 변경 후 장바구니 새로고침 실패도 미처리 rejection으로 남지 않도록 보강

## 5. 기대 동작

- 백엔드가 꺼져 있어도 앱 쉘과 공개 화면은 유지된다.
- 상단 레이아웃이 사라지지 않고, 사용자에게 서버 연결 실패 메시지가 보인다.
- 로그인/장바구니 같은 보호 기능은 기존처럼 세션 상태에 따라 동작한다.

## 6. 테스트

- 추가 테스트: `src/test/session-shell.test.tsx`
- 검증 포인트
  - 초기 세션 조회 실패 시 `AppShell`이 계속 렌더되는지
  - 사용자에게 백엔드 연결 실패 메시지가 보이는지
  - 로딩 칩이 해제되는지

## 7. 남은 제약

- 공개 데이터 API 자체가 모두 실패하면 홈/상품/커뮤니티는 각 화면의 오류 메시지를 보여준다.
- 이번 변경은 "화면 전체 소실"을 막는 안정화 작업이며, 오프라인 캐시나 재시도 전략까지는 포함하지 않는다.
