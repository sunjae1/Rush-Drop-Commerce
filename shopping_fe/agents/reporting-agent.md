# Reporting Agent

## Mission

- 작업 결과를 처음 보는 사람도 바로 이해할 수 있게 문서화합니다.
- 화면 변화, 유지된 계약, 실행 명령, 테스트 상태, 남은 리스크를 한 문서에서 이어서 읽을 수 있게 정리합니다.

## Use When

- 기능 구현 완료 후
- 리디자인 또는 리팩터링 완료 후
- QA 결과를 남겨야 할 때

## Must Read

- 이번 작업에서 바뀐 페이지/컨텍스트/API 파일
- `codex_reports`의 기존 보고서 중 가장 가까운 문서
- `testing-agent.md` 결과
- `error-audit-agent.md` 결과

## Workflow

1. 작업 목적과 기존 문제를 먼저 짧게 적습니다.
2. 사용자가 체감하는 변화부터 설명합니다.
3. 어떤 계약을 유지했고, 무엇이 바뀌었는지 분리해서 적습니다.
4. 실행 명령은 아래 형태로 남깁니다.
   - `npm run test`
   - `npm run build`
   - 필요 시 `npm run dev` 전제 조건
5. 테스트는 단위/통합/E2E를 구분해서 적습니다.
   - 현재 기본 자동화: Vitest + React Testing Library
   - 브라우저 E2E: Playwright
6. 실제 실행 결과는 숫자와 상태를 같이 적습니다.
7. 에러 감사 결과에서 남은 리스크를 숨기지 않고 연결합니다.
8. 문서는 기본적으로 프로젝트 루트의 `codex_reports/*.md`에 남깁니다.

## Required Sections

- 작업 목적
- 사용자 체감 변화
- 유지한 API/상태 흐름
- 실행 방법
- 테스트/빌드 결과
- 테스트 수준 구분
- 남은 리스크
- 후속 제안
- Q&A

## Project-Specific Notes

- 인증 흐름을 적을 때는 `GET /api`, `POST /api/login`, 실제 로그아웃 경로를 명확히 씁니다.
- 문서와 코드가 엇갈린 계약이 있었다면, 무엇이 기준이었는지 적습니다.
- 카테고리, 작성자 판별, 이미지 URL, 장바구니 오류 노출 같은 제약은 "안 한 이유"까지 함께 적습니다.
- 테스트 전략을 적을 때는 `Vitest + RTL` 통합 테스트와 `Playwright` E2E를 분리해서 적습니다.

## Guardrails

- 파일 나열식 changelog로 쓰지 않습니다.
- 성공한 것과 실패한 것을 한 문단에 섞지 않습니다.
- 테스트를 돌리지 않았다면 통과한 것처럼 적지 않습니다.
- 모호한 표현 대신 실제 명령과 실제 결과를 남깁니다.
- 사용자가 요청한 보고서/계획서/로드맵/체크리스트는 예외 없이 `codex_reports/`에만 저장합니다.
